## 1. Как машина учится (ML-фундамент)

**Главный переворот ML:** в обычной программе человек пишет правила (if-then). В ML машина выводит правила сама из множества примеров.

**Модель** — функция с настраиваемыми «ручками». Метафора: коробка с миллиардами ручек — покрутишь так, ошибается; эдак — угадывает.

**Параметры (веса)** — сами ручки. «Модель на 70B» = 70 млрд параметров. Больше ручек → потенциально точнее, но дороже.

**Обучение** — настройка ручек по примерам:
показали пример → модель ответила → посчитали ошибку (**loss**) → подкрутили ручки туда, где loss меньше → повторили миллионы раз.

**Градиентный спуск** — алгоритм выбора направления «вниз» по ошибке. Метафора: спуск в долину в тумане — видишь только под ногами, шагаешь туда, где склон вниз.

**Learning rate** — размер шага. Слишком большой — перепрыгнешь долину; слишком маленький — будешь спускаться вечно.

**Три способа обучения:**
- Supervised (с учителем) — примеры с готовыми ответами.
- Unsupervised (без учителя) — модель сама ищет структуру.
- Reinforcement learning (RL, с подкреплением) — учится через награду, методом проб.

## 2. Лесенка архитектур: нейросеть → LLM

Каждая ступень = предыдущая + уточнение. Матрёшка.

- **Нейросеть** — ручки, организованные слоями (этапы обработки в цепочку).
- **Слой** — группа ручек = один этап; каждый следующий работает с результатом предыдущего. Понимание копится: штрихи → формы → лицо.
- **Deep learning** — нейросеть из многих слоёв. «Глубокое» = много слоёв, никакой мистики.
- **Трансформер** — вид глубокой сети (2017), фишка — механизм внимания. Буква T в GPT.
- **LLM** — трансформер + текст + масштаб + задача «угадай токен».

Метафора: трансформер — чертёж двигателя (тип устройства), LLM — конкретный двигатель, заправленный и поставленный в машину (применение).

Другие архитектуры: **CNN** (свёрточные — картинки), **RNN/LSTM** (предшественники трансформера для текста; читали по очереди, забывали начало), **диффузионные** (генерация картинок из шума).

## 3. Как работает LLM

### Токены
- Модель видит не буквы и не слова, а **токены** — кусочки текста.
- Частое слово ≈ 1 токен; редкое длинное рубится на части.
- 1 токен ≈ 0,75 английского слова. Русский «дороже»: 2–3 токена на слово, потому что **токенизатор обучен преимущественно на английском**.
- Следствия: деньги и лимиты считаются в токенах; «глупые» ошибки типа «сколько букв р» — модель не видит букв (метафора: посчитать пиксели в картинке, на которую смотришь как на целое).

### Путь одного токена сквозь модель
**эмбеддинг → десятки слоёв (внимание + feed-forward) → вероятности → выбор**

- **Эмбеддинг** — токен превращается в вектор чисел, где смысл = геометрия. Классика: король − мужчина + женщина ≈ королева.
- **Внимание (attention)** — каждый токен смотрит на остальные с разной силой и подмешивает их смысл. «Сел» смотрит на «кот», чтобы понять, кто сел.
- **Feed-forward** — блок «обдумывания» после внимания: модель достаёт знания из весов про собранное.
- **Слой трансформера** = пара «внимание + feed-forward», повторяется 80–100 раз.
- **Вероятности** — финальное представление сравнивается со всеми ~100k токенами словаря.
- **Выбор** — с долей случайности. **Temperature**: высокая = мягче/разнообразнее, низкая = жёстче/предсказуемее.
- Весь путь прогоняется ЗАНОВО на каждый токен → вот почему ответ набирается по словам.

**Авторегрессия** — генерация по одному токену, каждый раз перечитывая весь текст. «Пишет, перечитывая себя».

### Эмерджентность (ядро всего)
Рассуждение, знания, код никто не вкладывал — они **возникли сами**, потому что оказались необходимы для хорошего угадывания токена на огромном масштабе. Чтобы дописать «2+2=», надо уметь считать; «убийцей оказался…» — держать логику сюжета. Сложное самозародилось из массы простого.

### Галлюцинации
Модель оптимизирована на **правдоподобное**, а не **правдивое** продолжение. Обычно совпадает, но не всегда. Не баг, а природа. Лечится не «починкой модели», а системой вокруг: RAG, инструменты, ссылки на источники.

### Беспамятность и контекст
- Модель не помнит НИЧЕГО между обращениями.
- Иллюзия памяти: каждый раз ей заново скармливают весь разговор целиком.
- **Контекст** — весь текст, поданный на вход прямо сейчас. **Контекстное окно** — его потолок (например, 200k токенов). Вылез за предел — старое выпадает.
- «Память» продуктов — внешняя надстройка: система хранит факты и подкладывает в контекст.
- **RAG** — не пихать всю базу в окно, а доставать релевантные куски по запросу.

### Три ступени обучения ассистента
**Pretraining → Finetuning → RLHF**
- Pretraining — угадывать токен на всём интернете. Набирает знания и язык.
- Finetuning — дообучение на чистых примерах «как отвечать». Учится быть полезным.
- RLHF — RL с наградой от людей: показывают ответы, люди выбирают лучший, модель учится предпочтениям.

### Идеи Карпаты
- **LLM OS** — метафора: LLM как процессор (CPU) операционной системы; context window = RAM; файлы+эмбеддинги (RAG) = диск; инструменты (Calculator, Python, Terminal) = периферия; browser = сеть; other LLMs = связь агентов.
- **Software 1.0 vs 2.0** — код-правила от человека vs правила, выученные сетью из данных.
- **System 1 / System 2** — быстрое автоматическое vs медленное пошаговое мышление (reasoning-модели).

## 4. Tool use, агенты, харнесс

### Tool use — как мозгу дают руки
Голая модель умеет только текст→текст. Гений в комнате без дверей.

**Ключевой механизм:** модель сама НИЧЕГО не делает — она только генерирует *запрос на вызов* инструмента. Реальный вызов исполняет обвязка (код вокруг). «Модель просит — система решает и исполняет». На этом стоит вся безопасность.

**Инструмент = две стороны:**
- Описание (меню) — имя, зачем, аргументы. Это видит модель. Код за ним — нет.
- Реализация (кухня) — реальный код. Исполняет обвязка.
Метафора: меню в ресторане — гость называет блюдо, кухня готовит по рецепту, гость на кухню не заходит.

Результат вызова кладётся в контекст — иначе беспамятная модель его не увидит.

**Три слоя:** модель (мозг) → обвязка/харнесс (ловит запросы, исполняет) → внешний мир (данные, сервисы).

**Принцип границ:** ИИ ограничен ровно своими инструментами — и это не баг, а способ управления. Не дал инструмент → действие невозможно. «Завязано на коде» = «подконтрольно».

### Агент
**Агент = LLM в петле + инструменты + цель.**
- Чат — линия: вопрос → ответ → стоп.
- Агент — кольцо: подумал → вызвал инструмент → проверил → не готово? ещё круг → готово? выход.
- Петлю крутит обвязка, но судьбу каждого круга решает модель: попросила инструмент = ещё круг; выдала текстовый ответ = выход.

**Две проблемы петли:**
1. Разваливается на длинных задачах — контекст раздувается, модель теряет нить, забывает, ходит по кругу.
2. Плохо решает, когда остановиться — рано бросает / крутится вечно / делает лишнее.

### Управление поведением — от совета к принуждению
| Механизм | Сила | Модель может проигнорировать? |
|---|---|---|
| Правила (CLAUDE.md) | мягкая — текст в контексте | да, иногда |
| Навыки (skills) | средняя — подгружаются по релевантности | может не подтянуть |
| Хуки (hooks) | жёсткая — код на событие, снаружи модели | нет |

Метафора: правила — устав (всегда помнишь), навыки — справочники на полке (берёшь по нужде), хуки — физическая блокировка на станке.

**Принцип:** чем необратимее и опаснее действие, тем дальше контроль от «модель попросим» к «система не позволит». Критичные двери закрывают хуком, не уговором.

### Изолированность агентов
Изоляция = своё отдельное пространство: свой контекст + свои права.
- Изоляция контекста — «своя память, свой стол» (субагенты со своим окном).
- Изоляция прав — «своя связка ключей» (code-reviewer с Read, но без Write).

Зачем: (1) чистый контекст — не захлёбывается; (2) ограниченный урон — безопасность (.env deny-read); (3) лёгкая отладка — знаешь, где искать; (4) параллельность — не мешают друг другу.
Цена: агентам сложнее обмениваться, надо передавать явно. Тот же компромисс, что монолит vs микросервисы.

### Две ветки — строить vs работать
| | Ветка 1 — свой продукт | Ветка 2 — Claude Code |
|---|---|---|
| Что это | твой код обращается к модели (API) | агент пишет код за тебя |
| Чья обвязка | твоя | заводская (Anthropic) |
| Момент | продукт живёт в рантайме | стройка |

Веткой 2 строят Ветку 1. Метафора: строительные леса (В2) и дом (В1). Ловушка: «я поговорил с Claude — у меня ИИ-продукт» — нет, ты пользовался помощником.

### MCP и LSP
- **MCP (Model Context Protocol)** — стандартный «разъём» между моделями и инструментами. M+N вместо M×N. Метафора: USB.
- **LSP** — тот же приём раньше: стандарт между редакторами кода и языками. Паттерн: «стандартизируй интерфейс — убей комбинаторный взрыв».
- Три роли: автор протокола (Anthropic), **сервер** (выставляет инструменты), **клиент** (втыкается и пользуется).
- MCP не обязателен: инструмент можно дать напрямую («припаять к одному коду»). MCP окупается на масштабе: готовые чужие серверы, переиспользование, сборка рук из разных источников, обмен с сообществом.

### Харнесс (harness)
Вся обвязка вокруг модели, превращающая мозг в работающего агента:
| Часть | Что делает |
|---|---|
| Цикл выполнения | шаги: посмотрел → подумал → вызвал → проверил → дальше |
| Управление контекстом | что подложить модели (RAG) |
| Оркестрация инструментов | даёт и вызывает инструменты (MCP) |
| Правила | мягко направляют (CLAUDE.md) |
| Хуки | жёстко принуждают на событиях |
| Память | хранит факты между сессиями |

**Главный тезис:** модель у всех плюс-минус одна. Ценность и безопасность продукта живут в харнессе.

### Пять слоёв расширения Claude Code
CLAUDE.md (правила всегда) · Skills (процедуры иногда) · MCP (внешние системы) · Subagents (своя петля, свой контекст) · Hooks (автоматика на события).
Тест выбора: верно каждый ход → CLAUDE.md; нужно иногда → навык; должно срабатывать автоматически → хук.

Skill = папка с SKILL.md: YAML-фронтматтер (когда использовать) + markdown-инструкции (как). ~100 токенов пока не активирован — progressive disclosure.

## 5. Практика: запуск харнесса с нуля (прожито руками)

### CLI и терминал
**CLI** — управление программой текстовыми командами (vs GUI с кнопками). Агенты живут в терминале, потому что сами текстовые.
Prompt показывает: кто ты @ на какой машине % в какой папке. Команды: `cd` (перейти), `ls` (список), `cat` (показать файл), `~` (домашняя папка).

### Путь запуска проекта
1. Проверка окружения: `git --version`, `node --version`, `pnpm --version`.
2. `git clone <url>` — скачать репозиторий.
3. `pnpm install` — поставить зависимости (заодно поднялись git-хуки: гейт briefs-immutable).
4. Запуск pipeline, превью через `pnpm dev` → `localhost:3000/...`.

### localhost и порты
- **localhost** — твой компьютер, обращающийся сам к себе. «Личный мини-интернет», виден только тебе. Метафора: кухня, где пробуешь блюдо до подачи в зал.
- **Порт** — номер «двери» (localhost:3000). Один сервер = одна дверь. `EADDRINUSE` = дверь занята (сервер уже запущен).
- 3000 — конвенция Next.js/Node. Разные проекты = разные порты; разные страницы одного проекта = разные пути (/landings/crm).

### Петля агента вживую (фазы pipeline)
- **Routing** — детерминированная развилка на входе (без LLM!): домен покрыт → PHASED, нет → STOP с todo.
- Фазы двух типов: детерминированные (код, пролетают тихо: P0, P3) и LLM-фазы (мозг думает, спрашивает разрешения: P1, P2, P4…).
- Каждая фаза пишет артефакт в файл → **чекпойнты**: обрыв (лимит, ошибка) не теряет прогресс, rerun продолжает с места остановки.
- **Repair-loop** — валидатор нашёл ошибку → фаза чинится, max 3 попытки → эскалация.
- **Schema (JSON Schema)** — жёсткая форма ответа для LLM-фазы: творчество внутри, форма снаружи. Намордник на галлюцинации.
- Откуда мозг «знает»: (1) задание-промпт от обвязки (с контекстом прошлых фаз), (2) файлы проекта, которые он читает (wiki/), (3) рассуждение самой модели. Знание — снаружи, мышление — внутри. → Качество выхода зависит от качества файлов в wiki/.

### Гейты
- **Гейт** — проверка-ворота: не прошёл — дальше не пускают. Hard (блокирует) vs soft (предупреждает).
- **CLI-гейт** — проверка вшита ВНУТРЬ команды-инструмента (не снаружи, как хук). Работает под любым мозгом → провайдер-нейтральная защита.
- Трёхслойная защита: git pre-commit + CLI-гейты + Claude-хуки. Принцип: критичное правило живёт там, где его нельзя обойти сменой мозга.

### Git-безопасность при работе агента
- `git add` — сложить в коробку (безопасно) → `commit` — запечатать локально → `push` — отправить всем (ВОТ ТУТ притормози).
- Смотри, куда агент пишет: .context/ и content/landings/ — ок; briefs (IMMUTABLE) и generated/ руками — красный флаг.
- «Yes» лучше «Yes, don't ask again» для git-операций — держи контроль.

## 6. Продакт-инженерия: метод диагностики

### Золотые правила
- **Чини источник, а не выход.** Плохой текст в лендинге → не править лендинг (перегенерится), а найти фазу-источник и её файл (wiki, промпт, правило).
- Вопрос диагностики: «на какой фазе это решилось и откуда фаза взяла решение?»
- «Некачественно» — не диагноз, а симптом. Разложи на конкретные контуры (тон? домен? разнообразие? структура?) — у каждого свой валидатор и адрес.

### Формат инженерной задачи
**Симптом → Гипотезы (H1, H2...) → Что проверить по каждой → Решение при подтверждении → Порядок действий.**
Дешёвая проверка (микро-тест) убедительнее теоретического спора.

### Четыре рычага улучшения (от мягкого к жёсткому)
1. Знания (wiki/, reference-файлы) — что мозг читает.
2. Промпт фазы — что мозг просят.
3. Правила (CLAUDE.md, редполитика) — как себя вести всегда.
4. Валидатор/гейт (код) — что физически не пропустить.
Чем критичнее и чаще проблема — тем правее рычаг.

### Ключевые понятия из реальных кейсов
- **Scope** — границы того, на что система рассчитана. «Вне скоупа» = за границей назначения. Scope creep — расползание. Кейс: завод строен под лендинги, статью-подборку запросили вне scope → закономерный провал, не баг.
- **Фронтматтер как улика:** `status: planned` — автор сам пометил фичу как недостроенную. Дешёвое однозначное доказательство, снимает споры.
- **Правило не заменяет инструмент:** «бери факты с сайтов» в ТЗ бесполезно, если у агента нет веб-инструмента. Нет рук — не дотянется.
- **Рассинхрон версий:** улучшения живут локально у людей, не слиты в общий main → у каждого своя версия, результат непредсказуем. Чинится процессом: единый source of truth + PR-дисциплина.
- **Богатый вход vs бедный:** в чате человек «разогревает» модель диалогом; в заводе вход = бриф. Если бриф беднее живого промпта — выход беднее. Качество входа = качество выхода.
- Продуктовый вопрос применимости: если чат стабильно лучше завода — либо завод недонастроен, либо задача не для завода. Ценность харнесса = повторяемость, правила, контроль, масштаб (не разовые задачи).

## 7. Классическая разработка

### Фронтенд / бэкенд
- **Фронтенд** — «зал»: всё, что видит и трогает пользователь (интерфейс).
- **Бэкенд** — «кухня»: логика, данные, вычисления за кулисами.
- Соединяет **API** — «официант» между ними.
- Диагностика «сохранил, но пропало»: перезагрузи страницу. Данные появились → фронт не обновился; пусто → бэк не сохранил.

### API
- **Методы (глаголы):** GET — покажи (только читает, безопасный); POST — создай; PUT/PATCH — измени (целиком/частично); DELETE — удали (опасный, необратимый).
- **CRUD** = Create, Read, Update, Delete.
- **REST** — договорённость, как строить API: сущности = ресурсы по адресам, действия = методы. `DELETE /orders/7` — адрес говорит С ЧЕМ, метод — ЧТО. Альтернатива: GraphQL.
- **Производительность:** кэширование (копия популярного под рукой), пагинация (данные порциями-страницами, как лента соцсети), асинхронная обработка (долгое в фоне; метафора: химчистка — сдал и ушёл, не стоишь ждёшь).

### Базы данных
- БД — «кладовая»: постоянная память продукта.
- **SQL (реляционные)** — строгие связанные таблицы-картотека: PostgreSQL, MySQL. Для точного и связанного (банк, заказы).
- **NoSQL** — гибкие: MongoDB, Redis. Для разнородного и масштабного.
- **Postgres + Redis** — классика: надёжный главный склад (SQL, диск) + быстрый кэш популярного (в памяти). Почему не всё в Redis: RAM дорогая и временная.

### Архитектура
- **Монолит** — весь код одна программа. Проще на старте; при росте: деплой целиком, баг роняет всё, команде тесно.
- **Микросервисы** — независимые части, общаются через API. Гибкость, отказоустойчивость, параллельные команды; но сложнее, дороже, труднее отладка. Метафора: ресторан vs фуд-корт.
- Путь: старт с монолита → рост → распил на сервисы. Преждевременные микросервисы — классическая ошибка.
- **Архитектура = скорость не сегодня, а через год.** Планировка дома: быстро въехать vs легко достраивать.

### Облако, стек, техдолг
- **Облако** — аренда чужих серверов (AWS, GCP, Azure, Яндекс). Метафора: такси vs своя машина. Главный плюс — масштабируемость по требованию.
- **Техстек** по слоям: язык (Python, JS/TS, Java, Swift/Kotlin) → фреймворк (React, Next.js, Django, FastAPI) → база (Postgres, Mongo) → инфраструктура (облако, Docker). Стек определяет: что дёшево/дорого, кого нанимать, что возможно.
- **Техдолг** — срезанные углы ради скорости. Как финансовый долг: взял быстро, платишь процентами (баги, замедление). Копить нельзя.
- **Рефакторинг** — переписать запутанное начисто, не меняя поведения. Уборка шкафа. Инвестиция в скорость будущей разработки. Продакт балансирует: сколько долга терпимо vs когда платить (считая косты).

### Процесс разработки
- **IDE** — рабочая среда разработчика (VS Code, JetBrains). Мастерская с инструментами.
- **Git-цепочка:** ветка (черновик) → push (выгрузил ветку на сервер) → PR (прошу влить) → code review (проверка коллег) → merge (влили в main-чистовик) → у всех.
- **Среды:** dev (песочница разработчиков) → stage (генеральная репетиция) → prod (боевая, пользователи). Театр: репетиционный зал → прогон → премьера. «Баг только на проде» = код тот же, обстановка другая (реальные данные, объём, нагрузка).
- **CI/CD:** CI (continuous integration) — автопроверки на каждое изменение (тесты, сборка); CD (delivery/deployment) — автодоставка на среды. Конвейер с контролёрами. «CI упал» = автопроверки нашли проблему и заблокировали merge — это защита, а не сбой.
- **Релиз** — выпуск версии пользователям. Веб — мгновенно (доска объявлений); мобилка — через магазины: сборка → проверка Apple/Google (дни) → пользователь должен обновиться (журнал: допечатать, развезти, купить). → В мобилке цена ошибки выше, релизы осторожнее.

## 8. DORA 2026: ROI of AI-assisted Software Development

### Центральный тезис
AI — **усилитель**: умножает сильные стороны здоровых организаций и дисфункции слабых. Наибольший ROI — не от инструментов, а от **системы под ними**: качество платформы, ясность процессов, слаженность команд. Без фундамента ИИ создаёт «локальные очаги продуктивности, теряющиеся в хаосе ниже по потоку». (= «дело не в модели, дело в харнессе» на уровне организации.)

### J-Curve реализации ценности (по первоисточнику)
Хорошо документированный феномен трансформаций: производительность **временно проседает**, пока команда учится, потом поднимается выше. Спад почти гарантирован; глубина пропорциональна масштабу изменения.

**Три системных фактора спада (имена из отчёта):**
1. **The learning curve** — время уходит на освоение инструментов и переход «от промптинга к системам на контексте, интенте и спецификации».
2. **The verification tax** — время на ревью сгенерированного кода (недоверие, галлюцинации) + сам объём кода растёт → бремя проверки расширяется.
3. **Pipeline adaptation** — даунстрим-процессы (тесты, согласования) должны масштабироваться под возросший объём.

**Почему инициативы гибнут:** не потому что технология плоха, а потому что руководство трактует фазу обучения как провал и **режет бюджет на дне ямы**.

**Рецепт DORA:** (1) явно бюджетировать «tuition cost» как стратегическую инвестицию; (2) automated testing + CI — ключи к развороту спада; (3) мониторить выход метриками: lead time for change, change failure rate — «система раннего предупреждения».

**Оговорка отчёта:** форма ямы (глубина, длительность) непредсказуема; зависит от технической зрелости, культуры обучения и здоровья платформ.

### Greenfield vs Brownfield
- **Greenfield** («зелёное поле») — проект с нуля, чистый лист. ИИ даёт 35–40% прироста: генерировать новое — его стихия, нет legacy-контекста.
- **Brownfield** — работа внутри существующего сложного кода. 10% или меньше: систему не охватить контекстом, легко сломать связанное, неявные договорённости не в коде.
- Правило: greenfield-задачи ИИ смело; к brownfield — осторожно (сначала достроить систему / расширить scope).
- Метафора: строить новый дом vs перестраивать старый с жильцами.

### Прочее из отчёта
- **Instability tax:** ИИ повышает индивидуальную эффективность, но растёт нестабильность доставки (пример: change failure rate 5%→6% дал −$344k в их калькуляторе).
- **Reinvest capacity, don't cut headcount** — выгода в высвобождении мощности, не в сокращениях.
- Считать ROI тремя сценариями: консервативный / реалистичный / оптимистичный.

### Критика отчёта (читать скептически)
ROI-модель смешивает выгоды: инвестиции в платформу/процессы/тесты улучшали доставку и ДО ИИ; отчёт не изолирует добавочную выгоду именно ИИ. Плюс коммерческий интерес (Google Cloud). → Навык: отделять «ИИ помог» от «мы навели порядок в процессах».

## 9. Разное

- **Стрим** — контекстозависимое слово: (1) streaming данных — передача потоком по кусочкам, не дожидаясь целого (ответ LLM токен за токеном, видео на YouTube); (2) workstream — рабочее направление в проекте; (3) трансляция.
- **Роутинг** — выбор маршрута: развилка, где система решает, по какому пути пустить задачу (навигатор).
- **Выбор модели = product/cost-решение:** не «всегда самую мощную», а под задачу. Структурные фазы — Sonnet; творческий текст — можно Opus. Мощная модель на простую фазу = переплата без выигрыша.
