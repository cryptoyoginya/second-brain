# MCP и LSP

Один паттерн, применённый дважды: **стандартизируй интерфейс — убей комбинаторный взрыв.** MCP делает это для «модели ↔ инструменты», LSP сделал раньше для «редакторы ↔ языки».

> [!quote]
> M×N разъёмов превращаются в M+N. Метафора — USB: один стандарт, и любое устройство втыкается в любой порт.

## Суть
- **MCP (Model Context Protocol)** — стандартный «разъём» между моделями и [[Tool use и границы агента|инструментами]]. Без него каждую модель пришлось бы отдельно подружить с каждым инструментом (M×N); со стандартом — M+N.
- **LSP (Language Server Protocol)** — тот же приём раньше: стандарт между редакторами кода и языками. Раньше каждый редактор × каждый язык; после — один протокол.
- **Паттерн:** ввёл общий интерфейс — и комбинаторный взрыв интеграций схлопнулся.

## Подробнее
- Три роли в MCP: автор протокола (Anthropic), **сервер** (выставляет инструменты наружу), **клиент** (втыкается и пользуется). Пример: любой второй мозг/сервис поднимает MCP-сервер (`kb_search` и т.п.), а агент-клиент подключает его и читает чужую базу.
- **MCP не обязателен:** инструмент можно дать модели напрямую, «припаяв к одному коду». MCP окупается *на масштабе*: готовые чужие серверы, переиспользование, сборка «рук» из разных источников, обмен с сообществом.
- Связь с [[ИИ как инфраструктура]]: MCP — часть обвязки, превращающей ИИ во встроенную инфраструктуру, а не разовую интеграцию.

> [!tip] Что запомнить
> MCP = USB между моделями и инструментами; LSP — то же для редакторов и языков. Паттерн: стандартизуй интерфейс, и M×N интеграций станут M+N. MCP не обязателен, но окупается на масштабе.

> [!check] Практический вывод
> Один-два инструмента для одного продукта — можно напрямую, без MCP. Много инструментов, переиспользование между продуктами, чужие интеграции — заводи MCP-сервер. Вопрос не «модно ли», а «есть ли комбинаторный взрыв, который стандарт схлопнет».
