# Фронтенд, бэкенд, API

Базовая анатомия любого продукта: **фронтенд** — то, что видит пользователь; **бэкенд** — логика и данные за кулисами; **API** — официант между ними.

> [!quote]
> Фронтенд — зал ресторана. Бэкенд — кухня. API — официант, который носит заказы туда-сюда.

## Суть
- **Фронтенд** («зал») — всё, что видит и трогает пользователь: интерфейс, кнопки, экраны.
- **Бэкенд** («кухня») — логика, данные, вычисления, которых пользователь не видит.
- **API** («официант») — как фронт и бэк общаются: фронт просит, бэк отвечает.
- Диагностика «сохранил, но пропало»: перезагрузи страницу. Данные появились → фронт не обновился; пусто → бэк не сохранил. Дешёвый способ понять, на чьей стороне баг.

## Подробнее
**API-методы (глаголы):**
- **GET** — покажи (только читает, безопасный).
- **POST** — создай.
- **PUT / PATCH** — измени (целиком / частично).
- **DELETE** — удали (опасный, необратимый).
Это **CRUD** = Create, Read, Update, Delete.

**REST** — договорённость, как строить API: сущности = ресурсы по адресам, действия = методы. `DELETE /orders/7` — адрес говорит *с чем*, метод — *что*. Альтернатива — GraphQL.

**Производительность API:** кэширование (копия популярного под рукой), пагинация (данные порциями-страницами, как лента соцсети), асинхронная обработка (долгое — в фон; метафора: химчистка — сдал и ушёл, не стоишь ждёшь).

> [!tip] Что запомнить
> Фронт (зал) ↔ API (официант) ↔ бэк (кухня). Методы-глаголы GET/POST/PUT/DELETE = CRUD. REST: адрес — с чем, метод — что. «Сохранил, но пропало» — перезагрузка покажет, фронт или бэк.

> [!check] Практический вывод
> Понимание слоёв ускоряет диагностику и разговор с разработчиками: локализуй баг (фронт/API/бэк) до того, как заводить задачу. GET безопасен, DELETE необратим — это и про [[Git-безопасность при работе агента|осторожность]] к разрушающим действиям.
