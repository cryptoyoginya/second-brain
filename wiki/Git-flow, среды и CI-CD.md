# Git-flow, среды и CI-CD

Как код доезжает от разработчика до пользователя: цепочка git → прогон по средам → автоматика CI/CD. Театр: репетиция → прогон → премьера.

> [!quote]
> «Баг только на проде» = код тот же, обстановка другая: реальные данные, объём, нагрузка.

## Суть
- **Git-цепочка:** ветка (черновик) → push (выгрузил ветку на сервер) → PR (прошу влить) → code review (проверка коллег) → merge (влили в main-чистовик) → у всех.
- **Среды:** dev (песочница разработчиков) → stage (генеральная репетиция) → prod (боевая, пользователи). Театр: репетиционный зал → прогон → премьера.
- **CI/CD:** CI (continuous integration) — автопроверки на каждое изменение (тесты, сборка); CD (delivery/deployment) — автодоставка на среды. Конвейер с контролёрами.

## Подробнее
- **IDE** — рабочая среда разработчика (VS Code, JetBrains), мастерская с инструментами. Там и живёт git-цепочка.
- **«CI упал»** = автопроверки нашли проблему и заблокировали merge. Это защита, а не сбой — [[Фазы пайплайна и гейты|гейт]] на пути в main.
- **«Баг только на проде»** — код тот же, отличается *обстановка*: реальные данные, объём, нагрузка. Поэтому и нужны dev/stage — ловить до премьеры.
- Прямая связь с [[Чини источник, а не выход|дисциплиной source of truth]]: main — единый чистовик; [[Богатый и бедный вход|рассинхрон версий]] (улучшения живут локально, не влиты) чинится именно PR-дисциплиной.

> [!tip] Что запомнить
> Ветка → push → PR → review → merge. Среды: dev → stage → prod (репетиция → прогон → премьера). CI/CD — автопроверки и автодоставка; «CI упал» = защита сработала. «Только на проде» — дело в обстановке, не в коде.

> [!check] Практический вывод
> Требуй PR-дисциплину и единый main как source of truth — иначе у каждого своя версия и непредсказуемый результат. Не воспринимай «красный CI» как аварию: это гейт, который бережёт prod. Прогоняй через stage то, где важны реальные данные и нагрузка.
